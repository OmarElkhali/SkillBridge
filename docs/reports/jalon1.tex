\section{Introduction}
Dans le cadre du premier jalon,il nous est demandé de définir le domaine du projet, de présenter le problème traité, de proposer une architecture globale, de réaliser une modélisation du système et de justifier les choix technologiques. Le livrable attendu est un mini-rapport de cadrage et de conception. 

Le domaine retenu dans notre cas est \textbf{l'éducation}. Ce choix répond à une difficulté réelle rencontrée par de nombreux étudiants et débutants : ils disposent souvent d'une idée de projet, mais ne savent pas toujours quelles notions apprendre ni quels cours suivre pour progresser de manière structurée.

Pour répondre à ce besoin, nous proposons \textbf{SkillBridge}, une plateforme éducative qui part d'une idée de projet saisie par l'utilisateur afin de l'orienter vers des cours pertinents. L'objectif n'est pas seulement d'afficher un catalogue de ressources, mais de faire le lien entre un projet concret, les compétences nécessaires et les contenus de formation correspondants.

\section{Domaine et problématique}
Le domaine choisi est celui de \textbf{l'éducation numérique}. Aujourd'hui, les ressources pédagogiques sont nombreuses : cours en ligne, tutoriels, plateformes spécialisées, contenus universitaires et professionnels. Cependant, cette abondance crée un autre problème : il devient difficile de sélectionner rapidement les ressources vraiment utiles selon un besoin précis.

Prenons le cas d'un étudiant qui souhaite réaliser une application web sécurisée, une plateforme e-commerce ou encore un projet Big Data. Dans la plupart des cas, il connaît le résultat qu'il veut atteindre, mais il ne possède pas toujours une vision claire des compétences à maîtriser ni du chemin d'apprentissage à suivre. Il peut alors perdre du temps à consulter des ressources dispersées, parfois redondantes, parfois trop avancées ou au contraire trop générales.

La problématique retenue est donc la suivante : \textbf{comment concevoir une plateforme éducative capable d'associer une idée de projet à des cours pertinents, tout en s'appuyant sur une application Spring sécurisée et un pipeline Big Data pour collecter, stocker et analyser les données ?}

\section{Acteurs et fonctionnalités principales}
Le système met en jeu deux acteurs principaux : \textbf{l'utilisateur} et \textbf{l'administrateur}.

L'utilisateur est le bénéficiaire principal de la plateforme. Il peut créer un compte, se connecter, rechercher des cours, consulter les ressources disponibles, soumettre une idée de projet, recevoir des recommandations, enregistrer des cours et suivre sa progression. L'administrateur intervient pour gérer le contenu global de la plateforme. Il peut ajouter, modifier ou supprimer des cours, organiser les catégories, gérer les fournisseurs de contenu et superviser les données générales du système.

Les fonctionnalités prévues à ce stade sont les suivantes :
\begin{itemize}
    \item authentification et gestion des rôles ;
    \item gestion des utilisateurs ;
    \item gestion des cours, catégories et fournisseurs ;
    \item saisie d'une idée de projet ;
    \item recherche de cours par mots-clés ;
    \item recommandation de cours adaptés ;
    \item enregistrement de cours favoris ;
    \item suivi de progression ;
    \item collecte des événements utilisateurs pour analyse Big Data.
\end{itemize}

\section{Architecture globale du système}
L'architecture générale de \textbf{SkillBridge} repose sur deux parties complémentaires : une \textbf{application web Spring Boot} et un \textbf{pipeline Big Data Hadoop}.

La première partie correspond à la couche applicative. L'interface utilisateur est réalisée avec \textbf{React}, ce qui permet de séparer clairement le frontend du backend. Les requêtes sont traitées côté serveur par \textbf{Spring MVC}, tandis que la logique métier est centralisée dans les services applicatifs. La persistance des données relationnelles est gérée via \textbf{Spring Data JPA} en lien avec une base \textbf{PostgreSQL}. La sécurité de l'application est assurée par \textbf{Spring Security}, avec authentification, contrôle d'accès et gestion des rôles. Dans une étape ultérieure, des mécanismes comme OAuth2 et JWT pourront être ajoutés pour renforcer le processus d'authentification, mais ils ne constituent pas ici le cœur de la modélisation métier.

La seconde partie correspond au pipeline Big Data. Les données structurées stockées dans la base relationnelle sont transférées en mode batch vers l'écosystème Hadoop à l'aide de \textbf{Sqoop}. En parallèle, les événements générés par l'application, comme les recherches, les consultations ou les interactions de l'utilisateur, sont collectés en continu via \textbf{Flume}. Les données sont ensuite stockées dans \textbf{HDFS}, tandis que \textbf{HBase} sert à conserver certaines informations analytiques ou résultats utiles à réinjecter rapidement dans l'application.

Le traitement repose sur trois approches complémentaires : \textbf{MapReduce} pour les agrégations batch, \textbf{Hive} pour les requêtes analytiques et \textbf{Python} pour des traitements ciblés comme le nettoyage de texte, l'extraction de mots-clés ou le calcul de scores simples. Dans cette première version, la recherche et la recommandation ne reposent pas sur du machine learning ; elles s'appuient plutôt sur des méthodes plus explicables, fondées sur les correspondances entre projets, compétences et cours, ainsi que sur les statistiques d'usage.

\section{Modélisation du système}
Pour la modélisation, nous avons retenu un \textbf{diagramme de classes UML}. Ce choix permet de représenter les principales entités métier ainsi que leurs relations, et constitue une base logique pour la future implémentation avec les entités JPA dans Spring Boot.

L'entité centrale est \textbf{User}, qui représente l'utilisateur de la plateforme. Chaque utilisateur possède un \textbf{Role}, ce qui permet de distinguer les droits d'un utilisateur simple de ceux d'un administrateur. Un utilisateur peut soumettre plusieurs \textbf{ProjectIdea}, enregistrer plusieurs \textbf{SavedCourse} et suivre son avancement dans différents cours à travers l'entité \textbf{Progress}.

L'entité \textbf{Course} représente un cours accessible depuis la plateforme. Chaque cours est rattaché à une \textbf{Category} pour le classement thématique, et à un \textbf{Provider}, qui représente l'organisme ou la source du contenu. Les notions ou compétences couvertes par un cours sont représentées par l'entité \textbf{Skill}.

Lorsqu'un utilisateur soumet une idée de projet, celle-ci peut être liée à plusieurs compétences. À partir de ce lien, le système peut produire une ou plusieurs \textbf{Recommendation}, orientées vers les cours les plus pertinents. Cette structure reste simple, claire et suffisamment extensible pour les prochaines phases du projet.

\section{Choix technologiques}
Les technologies retenues ont été choisies en fonction de la cohérence du projet et des exigences du module. \textbf{React} a été sélectionné pour le frontend afin de proposer une interface moderne, réactive et facilement connectable à une API REST. \textbf{Spring Boot} a été retenu pour le backend, car il permet de structurer l'application de manière claire et robuste. \textbf{Spring MVC} s'occupe de la gestion des contrôleurs et des requêtes HTTP, tandis que \textbf{Spring Data JPA} simplifie l'accès à la base relationnelle et le mapping objet--relationnel.

Pour la sécurité, \textbf{Spring Security} est un choix naturel puisqu'il permet de gérer l'authentification, l'autorisation et la protection des routes. La base \textbf{PostgreSQL} est utilisée pour stocker les données transactionnelles de la plateforme : utilisateurs, cours, catégories, projets, progression et historique.

Du côté Big Data, le pipeline repose sur les briques attendues dans le sujet. \textbf{Sqoop} sert à importer les données structurées depuis la base relationnelle. \textbf{Flume} permet de collecter les événements et logs en continu. \textbf{HDFS} est utilisé comme espace de stockage distribué pour les données massives, tandis que \textbf{HBase} sert à conserver certaines données analytiques ou index utiles à l'application. \textbf{MapReduce} est prévu pour les traitements batch, \textbf{Hive} pour les requêtes analytiques, et \textbf{Python} pour les traitements complémentaires liés à l'analyse textuelle et au calcul de scores.

\section{Conclusion}
Ce premier jalon a permis de poser les bases du projet \textbf{SkillBridge}, une plateforme éducative orientée projet qui a pour but d'aider l'utilisateur à identifier les cours les plus pertinents en fonction de son besoin. Le domaine choisi est l'éducation, et la problématique retenue répond à un besoin réel d'orientation pédagogique.

L'architecture globale proposée relie une application web Spring Boot sécurisée à un pipeline Big Data conforme aux technologies attendues. Le diagramme de classes fournit une représentation claire des principales entités métier et prépare la suite du développement. Enfin, les choix technologiques effectués sont cohérents à la fois avec les exigences du module et avec la logique fonctionnelle du projet.

Cette phase de cadrage et de conception constitue donc une base solide pour le deuxième jalon, qui portera sur l'implémentation de l'application JEE et de la couche de sécurité.
